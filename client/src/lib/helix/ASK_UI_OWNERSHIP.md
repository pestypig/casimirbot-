# Helix Ask UI Ownership Map

This map keeps the `HelixAskPill.tsx` split readable while the component is being reduced. New extractions should land in one of these owner modules only when the code is pure display, formatting, parsing, or projection logic with no React hooks, store writes, fetches, voice playback, route authority, terminal authority, or chat persistence.

`client/src/components/helix/ask-console/HelixAskConsoleState.ts` is the canonical recrown status contract. `HELIX_ASK_CONSOLE_RECROWN_PHASES` fixes the required order: stabilize the bridge, extract display owners, quarantine behavior-sensitive paths, build the minimal shell, replace the bridge after parity, then retire the legacy pill after proof. `HelixAskConsole` now enters through `HelixAskConsoleRuntimeShell`, and that shell defaults to the legacy bridge until side-by-side operator-surface parity is live-proven.

Bridge replacement is tracked explicitly by `bridgeReplacementProvenGates`, `bridgeReplacementOpenGates`, and `bridgeReplacementReady`. The proven gates record the parts already recrowned enough for the new shell to reuse, including entrypoint routing, named bridge quarantine, shell layout defaults, active-doc snapshot helpers, request-envelope helpers, latest-turn selectors, latest copy/debug/read-aloud control-target selection, final-answer projection, unclipped long-answer display, workstation trace projection, no-observation typed-failure preservation, calculator gateway trace parity, provider/model metadata projection, chat projection, chat persistence payload shaping, the minimal shell surface, minimal-shell submit-plan construction, minimal-shell optimistic turn start, the injected minimal-shell transport seam, minimal-shell stream-event recording, minimal-shell injected-turn completion/failure handling, the lazy backend-runner adapter, minimal-shell local turn rendering through recrowned final-answer display, minimal-shell structured workstation trace row rendering, minimal-shell latest control binding for copy/debug/read-aloud payloads, minimal-shell backend debug-export materialization for matching latest-turn refs, a runtime-shell seam that can select the minimal runtime without rendering `HelixAskPill`, and component smokes proving the runtime shell can submit through injected minimal transport, preserve the active `doc=...` context handoff, and render a completed turn without the legacy bridge. The open bridge-replacement gate is live operator-surface parity validation and `bridgeReplacementReady` is false; legacy `HelixAskPill` remains the default active runtime until that gate closes.

 `remainingBehaviorSensitivePaths` remains exposed in the status contract for the retirement phase: it records the original live surface that must be audited before deleting or hollowing out the legacy bridge.

Operator-surface parity is tracked separately by `operatorSurfaceParityItems`, `operatorSurfaceParityProvenItems`, `operatorSurfaceParityOpenItems`, and `operatorSurfaceParityReady`. Runtime parity is not enough by itself. The recrowned minimal shell has unit coverage for the tracked surface pieces, but live testing showed the default swap changed the day-to-day console experience before parity was accepted. Layout/dock behavior and top-of-console readability remain open, and `operatorSurfaceParityReady` is false.

Legacy behavior inventory is tracked by `legacyBehaviorClassifications`. The minimum complete runtime and operator surface chrome are classified as `used_must_move`; the current legacy bridge default is `used_temporary_adapter`; development fallbacks/trap doors remain `unknown_quarantined`; conflicting legacy runtime paths are `conflicting_remove_after_golden_path_proof`. Do not delete or hollow out `HelixAskPill` paths without moving the used items or proving a classification change.

The future shell target is tracked as `minimalRuntimeShellOwns` and `minimalRuntimeShellForbiddenOwnership`: the shell may coordinate the live console surface, but it must not own backend terminal authority, model/tool execution, private retries, a private agent runtime, final-prose tool-observation scraping, or duplicated backend route authority.

`HelixAskConsoleRuntimeShell` defaults to the legacy bridge, owns safe layout defaults tracked as `runtimeShellActiveOwnership`, and can still select the minimal runtime explicitly for side-by-side comparison. The shell must not statically import `HelixAskPill`, fetch, write clipboard, run TTS, or make route/terminal authority decisions; those behaviors either live behind tested minimal runtime owners or remain isolated behind the explicit legacy bridge.

`HelixAskMinimalRuntimeShell` is the replacement crown candidate. It composes the recrowned runtime layout, runtime picker, composer, runtime status line, latest-turn debug drawer, and visible-surface slots for recrowned goal/steering/supplement chrome with local draft/runtime-picker/debug state without importing `HelixAskLegacyRuntimeBridge` or `HelixAskPill`. It submits through the injected/backend runner, preserves the active docs-viewer URL handoff, records stream events, renders structured workstation trace rows from debug/transcript payloads, binds latest controls, materializes matching-turn debug export, and persists/hydrates chat sessions.

The tested minimal runtime shell is a replacement candidate, not the default path. The lazy legacy bridge remains the active day-to-day UI until live operator-surface parity is accepted.

`HelixAskMinimalRuntimeSubmitPlan.ts` is the pure submit-planning boundary for the inactive shell: it normalizes the draft through the recrowned submit-admission helper, reads the active docs-viewer path from a supplied desktop URL, and builds the request envelope. It must not fetch, start streams, write clipboard, run TTS, import the legacy bridge, or make terminal-authority decisions.

`HelixAskMinimalRuntimeLifecycle.ts` is the pure optimistic turn-start and injected-turn completion boundary for the inactive shell: it appends the "Reasoning in progress..." reply, marks the shell busy, records the active turn id and start time, creates the initial client live event, then can complete or fail that same local turn from an injected transport result/error. It must not fetch, start streams, write clipboard, run TTS, import the legacy bridge, or make terminal-authority decisions.

`HelixAskMinimalRuntimeTransport.ts` is the injected transport seam for the inactive shell: it builds the backend turn payload and calls a caller-supplied runner. The module must not import `runAskTurn`, `runAskTurnStream`, `HelixAskPill`, or the legacy bridge, and it must not call `fetch` directly. Real backend transport remains an open parity gate until the runner is wired and tested against the live bridge behavior.

`HelixAskMinimalRuntimeBackendRunner.ts` adapts the injected transport seam to the existing Ask backend API by lazy-loading `runAskTurnStream` and falling back to `runAskTurn`. It keeps API imports out of module initialization and remains inactive until the minimal shell replaces the legacy bridge.

`HelixAskMinimalRuntimeTurnList.tsx` is the inactive shell's local reply display owner: it projects minimal runtime replies into prompt/final-answer cards, using existing provider/model metadata display helpers and `HelixAskFinalAnswer`. It must not fetch, start streams, write clipboard, run TTS, materialize debug exports, infer workstation observations from final prose, import the legacy bridge, or make terminal-authority decisions.

`HelixAskMinimalRuntimeControls.ts` is the inactive shell's side-effect boundary for latest visible turn controls. It builds copy, debug-copy, and read-aloud payloads from the latest minimal runtime reply and provides browser clipboard/speech adapters. It must not fetch, start Ask turns, import the legacy bridge, read stores, execute tools, or make route/terminal-authority decisions. Backend debug-export materialization is delegated to `HelixAskMinimalRuntimeDebugExport.ts`.

`HelixAskMinimalRuntimeDebugExport.ts` is the inactive shell's minimal debug-export materialization boundary. It builds a latest-turn scoped backend debug-export request from the control payload, accepts only matching turn refs, falls back to the local scoped debug JSON on stale/missing/failing backend materialization, and owns the browser `fetch` call for `/api/agi/ask/turn/:turnId/debug-export`. It must not import `HelixAskPill`, start Ask turns, execute tools, read stores, scrape final prose, or make route/terminal-authority decisions.

`HelixAskMinimalRuntimeChatSession.ts` is the inactive shell's durable chat hydration helper: it projects existing `useAgiChatStore` context-session messages into minimal runtime replies. Store reads/writes live only in `HelixAskMinimalRuntimeShell`, where the runtime shell owns desktop chat/session persistence; display helpers must not import the chat store.

`HelixAskLegacyConsoleInventory.ts` is the live legacy-slicing tracker for the recrown goal. It records the current active path through `HelixAskLegacyRuntimeBridge -> HelixAskPill`, the `HelixAskPill.tsx` source snapshot, the render-slice line range, already-recrowned display slices, behavior-sensitive quarantined slices, and unknown trap-door slices. This file is not proof that bridge replacement is ready; it is the map that prevents future work from confusing the tiny bridge wrapper with the still-active 29k-line legacy implementation.

`HelixAskAttachmentCommit.ts` is the deterministic attachment commit-check owner for image/text attachment submit readiness, structured preview metadata, and stale/too-large/unsupported/error reason shaping. Attachment mutation, source admission, request-envelope construction, attachment payload authority, and turn submission stay in the legacy bridge.

`HelixAskVoiceTimelineBuildInfo.ts` is the deterministic voice timeline build-info projection owner for initial client build-info state plus `/version` success/error payload shaping. Browser/env reads, `/version` fetch and abort lifecycle, diagnostics publication, mic/STT runtime, playback runtime, and TTS stay in the legacy bridge.

## Still Quarantined In `HelixAskPill.tsx`

The following categories remain behavior-sensitive and should not be moved into display modules as part of presentation cleanup:

- Timeline entry creation, ordering, patching, filtering, and feed state. Unused timeline label constants should be deleted instead of preserved as new display owners.
- Legacy fallback and fallback reply construction.
- Transport and fetch calls, including Ask turns, live-source mail, goal sessions, and debug export endpoints.
- Stale completion and in-flight turn lifecycle.
- Route metadata, source admission, route authority, and terminal authority.
- Chat persistence and session merging.
- TTS, voice playback, voice diagnostics, mic capture, and audio graph behavior.
- Workstation dispatch, panel mutation, goal-session actions, and user-input request handling.
- Debug export authority, rendered-button export fallback, and clipboard/export bounding.

## Behavior-Sensitive Queue

These issues came from live Helix-mode UI testing and must be handled as behavior-goal work, not display-helper extraction. Do not move the named paths into non-React display modules unless behavior-preserving tests prove the move is safe.

### HASK-BSQ-001: Active document context handoff into Helix Ask requests

Evidence:

- Browser URL had `doc=docs/research/nhm2-current-status-whitepaper-2026-05-02.md`.
- UI visible focus included `docs-viewer` and `scientific-calculator`.
- Prompt using "current NHM2 whitepaper" failed with: "I need retrieval before finalizing this claim. I do not yet have grounded evidence references for it."
- The same prompt succeeds through the direct API when `doc_path` is supplied.
- The same UI prompt succeeds when the doc path is explicitly written in the prompt.

Likely boundary:

- `HelixAskPill` request construction is not consistently passing retained active docs-viewer context, doc path, or document evidence into Helix golden-path requests, especially for compound prompts.

Expected:

- When the UI is on a docs-viewer URL with `doc=...`, Helix-mode Ask request assembly should include the active doc path/context in the backend request envelope without requiring the user to restate the path.

Quarantine:

- Request-envelope assembly, active-doc context collection, workspace context snapshots, route metadata, source admission, and golden-path request transport.

### HASK-BSQ-002: Scoped Debug copy binds to stale prior turn

Evidence:

- After a complex Helix UI compound answer rendered visibly, clicking latest Debug copy copied an older calculator turn.
- Copied payload had marker mismatch, `terminal_artifact_kind = workstation_tool_evaluation`, and answer "3 + 5 = 8".
- The visible answer was the later compound answer.
- Therefore latest visible reply and debug-copy target diverged.

Likely boundary:

- `helix-ask-latest-debug-copy` or selected debug state still points to a cached prior debug payload or previous latest reply after complex/streamed golden-path turns.

Expected:

- For every rendered answer, Debug copy must bind to that reply's `turn_id` and `debug_export_ref`, not the last successful cached debug export.

Quarantine:

- Stream final projection, latest reply selection, selected debug state, debug-copy binding, rendered-button export fallback, and clipboard/export bounding.

### HASK-BSQ-003: Backend compound terminal authority is not the first suspect

Evidence:

- Direct Helix API returns `terminal_artifact_kind = compound_evidence_synthesis_answer`.
- Direct Helix API returns `final_answer_source = compound_evidence_synthesis_answer`.
- Direct Helix API returns `subgoals = 5` and `terminal_error_code = null`.
- UI explicit-doc compound prompt renders a compound answer.

Implication:

- Patch UI request context and debug binding first. Do not treat this as a backend compound failure or golden-path terminal-authority failure without new contrary evidence.

## Extraction Rule

Before moving a helper out of `HelixAskPill.tsx`, classify it:

- `presentation`: display rows, labels, class names, copy formatting, deterministic tokenization.
- `input parsing`: non-mutating parsing that only returns structured display/input data.
- `quarantined behavior`: any helper that reads/writes stores, calls fetch, starts/resumes turns, changes panels, owns voice/TTS, constructs fallbacks, or decides route/terminal authority.

Only the first two categories belong in the non-React owner modules during this cleanup. Quarantined behavior needs a separate behavior-goal patch with contract tests.
