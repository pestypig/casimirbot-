## Executive Verdict

**overallCongruenceStatus:** **mixed / partially congruent** — the repo has a real GR brick pipeline (BSSN evolution + derived ADM/York-style channels) and a real Natario-style "low expansion" classification surface, but major mechanism claims (Casimir pump, strobing, sector timing) are **reduced-order/proxy modeled** and appear only weakly coupled (or not coupled) to the GR-evolved metric/brick surfaces. The shift-plus-lapse branch exists and is internally self-described as diagnostic/reference-tier with explicit provenance controls, but it is not a "first-principles generalized solution" in the sense of a jointly-solved lapse+shift profile with matter closure.

**canonicalNHM2Status:** **implemented; classification and proof-surface exist; confidence limited by what is actually "metric-derived" vs "projected/constructed."** The repo explicitly encodes the Natario "no contraction/expansion" concept via divergence/expansion checks and a divergence-reduction/projection step for shift fields (Natario-like branch).

**casimirMechanismStatus:** **proxy_modeled / reduced-order advisory** — Casimir energy density and "dynamic amplification" are implemented as parametric formulas and multipliers; there is no first-principles QFT field solution or a demonstrable closure to the GR/matter evolution.

**sectorStrobingStatus:** **implemented as duty-cycle/time-averaging logic; weakly evidenced; not GR-coupled** — timing/strobing exist as parameters and audits, but the current authority story is "policy/ownership" rather than "field transport solved."

**shiftGeometryStatus:** **implemented as visualization/diagnostic; interpretive** — the repo produces explicit shift geometry render artifacts/audits, but these are explanatory overlays and do not by themselves prove transport or mechanism closure.

**shiftPlusLapseStatus:** **implemented_and_tested; reference-only/diagnostic-tier by contract and provenance discipline** — `warp.metric.T00.nhm2.shift_lapse` exists, emits a lapse summary, exports lapse-gradient and Eulerian-acceleration diagnostics, computes a wall-normal `beta_outward/alpha` safety summary, and computes a tiny cabin clock-split; it also explicitly detects float32 under-resolution and falls back to an analytic companion with provenance tags.

**precisionProvenanceStatus:** **strong for the shift-plus-lapse "mild" scenario; mixed elsewhere** — the repo explicitly models "raw brick vs analytic companion vs mixed-source effective" for lapse-gradient underflow scenarios and carries explicit source fields; this is both implemented and tested.

**renderPresentationStatus:** **documented + audited; enforcement partially evidenced** — render taxonomy and separation policies exist as research docs and "latest" audits, but whether all downstream renderers *enforce* the taxonomy as a hard gate is not proven solely by the presence of audits.

**sourceMechanismPolicyStatus:** **policy-coded / advisory; non-authoritative** — the repo contains explicit audits and memos asserting reduced-order maturity and non-authoritative status, and a promotion contract with blocked parity. What is less evidenced (from code alone) is that these policy objects truly gate readiness rather than being narrative.

**parityRouteStatus:** **policy "blocked"; mathematically unresolved** — parity is described as blocked by a derivation-class mismatch (direct metric vs reconstructed proxy). Current evidence is primarily in audit/memo surfaces.

**exemptionRouteStatus:** **bounded advisory-only; activated for a narrow non-authoritative claim subset** — the exemption route is now explicitly active only for bounded source annotation, mechanism context, and reduced-order comparison claims, while formula equivalence, authority, viability promotion, and cross-lane expansion remain blocked.

**mostImportantFalsifier:** **compute and compare "low-expansion" on the authoritative (brick/GR) surface rather than on projected/constructed shift fields** — if `div(beta)` and/or `theta = -trK` consistency fails tolerance on the actual evolved brick (or on a truly metric-derived adapter surface declared non-proxy), the "Natario-like low-expansion canonical NHM2" classification collapses inside the repo's own declared canonical checks.

**recommendedNextAction:** **promote a single "authoritative low-expansion proof gate" that is brick-native, not projection-native, and make the "artifact latest.json" availability unambiguous** — right now, tests and audits refer to `*-latest.json` artifacts, but availability/commitment is not robustly demonstrated from the repo snapshot alone; tighten this so an external auditor can reproduce without hidden generation steps.

## Falsification Paths

**Warp-family / Natario-like low-expansion classification** can be falsified *inside the repo's own declared framework* by any of the following concrete contradictions:

A brick-native computation of `div(beta)` or an equivalent expansion observable (computed on the evolved GR brick fields, with declared stencils/boundaries) exceeds the repo's declared tolerance while the canonical check still claims Natario-like satisfaction. The canonical checker currently bases `divBeta` on `pipeline.warp.metricAdapter.betaDiagnostics` if present, and on a tolerance from `natario_expansion_tolerance`. A mismatch between where `betaDiagnostics` is computed (projected shift, sampled field, or true evolved beta) and where "authoritative" Lane A is defined would falsify congruence between "Natario-like" as a physical statement and "Natario-like" as a repo label. Natario's construction explicitly ties "zero expansion" to the expansion of Eulerian observers and uses a divergenceless generating field for a volume-preserving warp; if repo checks are not meaningfully connected to that concept on authoritative surfaces, the label is misleading.

**Casimir/source mechanism modeling** is falsified as "congruently represented" if any repo narrative claims cross over from "reduced-order energy density model" to "first-principles Casimir field solution." The implemented code path is explicitly formulaic (baseline Casimir-like density plus multipliers like Q, spoiling factors, duty cycle), which is not a Maxwell+QFT boundary condition solve. A concrete falsifier is discovering a downstream "proof" or readiness gate that treats these proxy source values as if they were the stress-energy tensor used in GR evolution without clearly tagging them `proxy` / `reduced-order`. The strict/proxy machinery exists, but it must be used consistently.

**Sector strobing / timing / TS** interpretation is falsified by a policy contradiction: the timing authority audit declares required readiness timing fields (`tauLC_ms`, `tauPulse_ms`, `TS_ratio`) and claims `timingAuthorityClosed=true`, yet also shows critical timing fields like `TS` / `epsilon` / `isHomogenized` as **missing** and still reports `blockingFindings: none`. If the repo claims timing closure as a physical closure rather than a governance decision ("readiness not blocked by missing TS"), that is falsified by this artifact itself.

**Shift geometry visualization** is falsified as "solve-backed" if the visualization uses mixed/proxy sources without explicit tagging or if it presents constant/constructed β fields as if they were evolved metric outputs. The GR evolve brick explicitly can set β from a pipeline shift vector (potentially constant), including in the shift-plus-lapse init path. If a shift geometry panel implies spatially varying solved transport while the data source is a constant β assignment, the visualization becomes misleading.

**Shift-plus-lapse generalized branch** can be falsified as a meaningful "nontrivial lapse profile" if the applied `lapseSummary` is not actually used in the GR brick generation. The GR brick constructor explicitly resolves a `lapseSummary` from pipeline state and applies it pointwise (with support/taper) to `state.alpha`. If a future refactor breaks `resolveShiftLapseSummaryFromPipeline` / `applyShiftLapseSummaryToState`, the branch degenerates into unit-lapse. Conversely, it can be falsified as a "true GR-resolved cabin gravity" if the repo claims the brick numerically resolves the mild gradient: the test suite itself expects raw float32 channels to underflow to zero and mandates analytic companion substitution.

**Render/presentation trust** is falsified if certified/proof-adjacent render categories can be produced without taxonomy labels or if reference-only overlays can appear in proof lanes without clear badges. Render taxonomy and separation policies exist as documents and audits, but the absence of an explicit "hard gate" in the evidence reviewed here means this remains a risk.

## Precision / Provenance Audit

**Raw brick values (float32) are used** throughout the GR brick pipeline: evolution state channels are float32 arrays, the serialized brick is `r32f`, and derived channels like `alpha_grad_*` and `eulerian_accel_geom_*` are computed directly from those float32 fields.

**Analytic companion values (float64 analytic)** are used explicitly for the mild shift-plus-lapse cabin gradient case because the expected gradient magnitude is extremely small (test asserts ~5-6×10^-17 1/m), which under-resolves in float32 brick gradients across the discretized hull scale. The repo's test suite requires that raw brick diagnostics show zero for `alpha_grad_z` and `eulerian_accel_geom_mag` in that mild scenario while analytic companions are nonzero, and it requires the effective diagnostics to prefer the analytic companion source.

**Mixed-source policy is explicit and structured.** In cabin lapse diagnostics, `time-dilation-diagnostics.ts` determines whether to use brick samples (`gr_evolve_brick_alpha`, `gr_evolve_brick_eulerian_accel`) or analytic lapse evaluation (`analytic_lapse_summary_companion`) based on whether the brick provides resolvable differences and nonzero gradients; it records which source was used for centerline α, top/bottom α, and gravity gradient. This is one of the repo's strongest "do not overclaim" implementations: it anticipates numerical underflow and encodes provenance so downstream consumers can tell what happened.

**What remains numerically weak (known failure mode):** treating mild lapse-gradient fields as if they were brick-resolved is explicitly rejected by tests. The repo expects under-resolution to produce zeros in raw gradient channels and provides an analytic fallback and a `mixed_source_prefer_analytic_for_underflow` policy. If a renderer ignores the `source` tags and prints the "effective" values without their provenance, the viewer can easily misinterpret the field as resolved from the GR simulation.

**Policy closures substituting for parity/authority:** the strict/verification gate in `time-dilation-diagnostics.ts` treats some missing/`proxy` proof-pack fields as disqualifying for certified labels, and it also treats some `latticeMetricOnly` Natario cases as having special requirements (e.g. require `metric_t00_contract_ok`, `theta_metric_derived`, `qi_metric_derived`, `ts_metric_derived`). This is a governance choice that can fail closed even if some mathematical parity is absent; it's coherent as "label policy," but it does not itself create physical parity.

## Promotion Audit

**Currently promotion-blocked (by explicit policy artifacts):**
The mechanism/source lane is described as reduced-order and non-authoritative, with blocked promotion eligibility and a parity route deemed structurally blocked by derivation-class mismatch.

**Currently advisory-only / reference-only (by contracts/tests):**
The ADM gravity diagnostic lane is explicitly reference-only and not authoritative for readiness in the contract file, and this is asserted in tests. The shift-plus-lapse diagnostics are framed as diagnostic-tier and mixed-source, not a "certified solve," and are treated accordingly by provenance.

**What evidence would be required to promote the non-authoritative mechanism lane (parity route):**
A promotion of the parity route would require a demonstrated mapping (with tolerances and reproducible artifacts) between the reconstructed proxy (source/mechanism model) and the direct metric-derived quantities on **matched scenarios**. The repo's current policy asserts this is blocked due to derivation class difference; to falsify that block, you would need a concrete derivation that shows how to transform or constrain the proxy model so that it yields the same invariants/constraints as the direct metric path under a shared parameterization.

**What evidence would be required to widen the active exemption route beyond the bounded claim subsets:**
The currently active bounded exemptions already depend on (a) stable numeric behavior, (b) explicit provenance separation, and (c) a demonstrated "non-misleading" interpretation under the render taxonomy, such that users cannot confuse advisory outputs for proof-bearing outputs. To widen the route beyond those three bounded non-authoritative claims, the repo would need stronger downstream consumer enforcement evidence, explicit additional disclaimers/forbidden-inference coverage, and a new contract showing why the wider claim surface still avoids parity, authority, viability, and cross-lane overclaim.

## Missing Pieces

**Brick-native low-expansion proof:** The repo has a Natario canonical check structure, but the decisive missing piece for falsifiability is a **single authoritative computation** of "low expansion" (e.g. `div(beta)` or an equivalent expansion scalar) defined on the same authoritative surface that drives Lane A and the GR solver outputs, with declared stencil order/boundary conditions and reproducible tolerances. Right now, the check is structurally present but its grounding surface (projected shift vs evolved brick vs adapter field) is not proven from the audited subset.

**Mechanism-to-metric closure:** Casimir "pump," "tiles," "sector strobing," and amplification are implemented as formulas and multipliers, not as a field-theory-to-GR closure. The repo is internally honest *if* it treats these strictly as reduced-order/advisory; it is incomplete if it implies the GR brick's stress-energy is derived from a physically realized Casimir mechanism rather than from chosen/proxy source models.

**Artifact ground truth availability:** Tests and audits reference `artifacts/research/full-solve/*-latest.json` paths (including `nhm2-shift-plus-lapse-diagnostics-latest.json`), but the repo snapshot alone does not guarantee these artifacts are committed vs generated during CI/test runs. The test suite reads the artifact file from disk and asserts its contents, implying generation must occur earlier or that it is expected to exist in the workspace. This is a reproducibility gap for external auditors unless generation steps are explicit and deterministic.

**Transport context for shift geometry:** Shift geometry panels exist and are audited, but there is no demonstrated transport solution (e.g. null geodesic integration, bundle transport) used as the backbone of those visuals; in time dilation diagnostics, redshift is explicitly "reduced_order null transport" and can fall back to proxy/unavailable states.

**Hard enforcement of render taxonomy:** Documentation and audits exist, but a robust falsifiable enforcement point would be a test or runtime gate that refuses to publish a render artifact when category labels are missing or inconsistent with data provenance. Evidence reviewed here shows policy, not necessarily enforcement.
